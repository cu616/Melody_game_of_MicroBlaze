# 2026-06-03 VS1003B 慢速初始化与 UI 颜色修改

## 用户要求

- VS1003B 模块仍然无声，用户确认耳机插在 PHONE，不是 LINE IN。
- 用户准备离开/休息，希望自动继续排查并生成新版。
- 按键时的打击反馈小方块不要固定白色，改成与当前判定颜色一致。
- `READY` / `FINISH` 状态提示色改成白色，RGB LED 也改为白色。

## VS1003B 排查依据

- 用户反馈的 LED 状态为 `LED0` 常亮、`LED1` 闪烁、`LED8/LED9` 亮。
- 按当前 debug 映射，这说明：
  - `LED8` 亮：`XRST` 已拉高。
  - `LED9` 亮：`DREQ` 为高。
  - `LED1` 闪烁：`SCLK` 正在活动。
  - `LED0` 常亮：`MOSI` 多数时刻为高，不足以单独判定错误。
- 因此 FPGA 侧确实在尝试发送 SPI，下一步重点不是换歌曲，而是让初始化更贴近商家例程并降低时序风险。

## 本次 VS1003B 修改

文件：
- `Mini_IO.srcs/sources_1/new/rhythm_video_audio.v`

修改点：
- `vs1003b_mp3_rom_player` 状态机从 4-bit 状态扩展到 5-bit 状态。
- 新增 `ST_SOFT_RESET_WAIT`，在写入 `SCI_MODE=0x0804` 后显式等待约 2ms，再继续写后续寄存器。
- 新增 `ST_POST_INIT_WAIT`，在写入音量后显式等待约 1ms，再发送 4 个启动用 `0x00` 数据字节。
- 硬复位低电平保持从约 10ms 增加到约 100ms。
- 硬复位释放后的等待从约 20ms 增加到约 200ms，贴近商家 `Mp3Reset()` 的长等待策略。
- `SCI_CLOCKF` 从 `0x6000` 改成商家例程使用的 `0x9800`。
- `SCI_VOL` 从 `0x0000` 改成商家例程播放阶段常用的 `0x0A0A`，避免极端最大音量引入不确定性。
- SPI 分频从每半周期约 32 个 100MHz cycle 改成约 100 个 100MHz cycle，SCLK 约 500kHz，优先保证低速稳定通信。
- 继续使用商家例程自带 `MusicDataMP3.c` 转出的 `music/processed/vendor_vs1003b_musicdata_mp3.mem` 作为 ROM 数据源。

## 本次 UI 修改

文件：
- `Mini_IO.srcs/sources_1/new/rhythm_video_audio.v`

修改点：
- 判定线附近和底部按键反馈小方块不再固定白色。
- 当前判定为 `GOOD` 时，小方块为绿色。
- 当前判定为 `BAD` 时，小方块为蓝色。
- 当前判定为 `MISS` 时，小方块为红色。
- 无判定时，小方块仍为白色，便于普通按键反馈。
- 未选择歌曲的 `READY` 中央提示改为白色。
- `game_finished` 或未选择歌曲时，RGB LED 改为白色。

## 上板测试建议

- 下载 `Mini_IO.sdk/design_mb_wrapper_hw_platform_0/download.bit`。
- VS 模块耳机插 `PHONE`。
- 打开 `SW2` 测试 VS1003B 商家样例 MP3 播放。
- 由于初始化等待变长，打开 `SW2` 后请等待约 0.5 秒再判断是否有声。
- 若仍无声，但 `LED8/LED9` 亮且 `LED1` 闪，下一步建议实现 SCI 读回，把 `SCI_MODE` / `SCI_CLOCKF` / `SCI_VOL` 的读回值显示到 LED 或数码管，用来确认 VS 芯片是否真正接收了寄存器写入。
